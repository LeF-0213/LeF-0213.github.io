# Express란?

Express는 Node.js 환경에서 가장 널리 사용되는 웹 애플리케이션 프레임워크로, 서버를 쉽고 빠르게 구축할 수 있도록 다양한 기능을 제공한다. 기본적인 HTTP 모듈보다 훨씬 간단하게 라우팅 처리, 요청/응답 객체 관리, 미들웨어 설정, 정적 파일 제공, 템플릿 엔진 연결 등을 할 수 있어 개발 생산성을 크게 높여준다. 특히 RESTful API 구축에 최적화되어 있으며, 수많은 미들웨어와 플러그인 생태계를 통해 다양한 웹 서비스에 손쉽게 확장할 수 있는 것이 장점이다.

### 설치

```termianl
npm install express
```

### 앱 생성

```js
const express = require("express");
const app = express();
const port = 3000;
```

## 💻 서버 사이드 렌더링(Sever-Side rendering, SSR)

* 전통적인 MVC 웹 애플리케이션
* Node.js EJS 뷰 엔진을 쓰는 방식

### 동작 방식

> 1. 브라우저가 요청을 보냄     
> 2. 서버가 DB에서 정보를 조회      
> 3. EJS 탬플릿 + 데이터를 합쳐서 완성된 HTML을 만듦
> 4. 완성된 HTML을 브라우저로 전송      
> 5. 브라우저는 HTML 그려서 화면에 보여줌   

### 장점

> 구조가 단순   
> 빠르게 페이지 만들기 좋음

## 💻 클라이언트 사이드 렌더링(CSR, SPA)
* 프론트엔드와 백엔드를 분리한 아키텍처
* API 서버 + 프론트(순수 JS, React/Vue) 분리

### 동작 방식
> js 코드 안에서 fetch 또는 axios 등으로 Node.js 서버의 REST API를 호출       
> 프론트에서 JSON 데이터를 받아서 DOM을 직접 만들거나 React/Vue 컴포넌트로 렌더링   

### 장점

> 프론트와 백엔드가 역할이 명확하게 분리      
> API 재사용성이 높음(모바일 앱, 웹 서비스, 외부 공개 API 등으로 호가장하기 좋음)       
> 프론트 프레임워크(React/Vue 등)와 연동이 좋음

### 🔖 주요 차이점 요약

| 구분             | SSR (서버사이드 렌더링) | CSR (클라이언트 사이드 렌더링) |
|:-----------------:|----------------------|-----------------------------|
| 렌더링 주체       | 서버                 | 클라이언트(브라우저)       |
| 초기 전송 파일    | 콘텐츠가 채워진 완성된 HTML | 뼈대와 JavaScript 파일        |
| 초기 로딩 속도    | 빠름 (콘텐츠 바로 표시)    | 느림 (JS 로딩 및 실행 필요)  |
| 페이지 이동 속도  | 느림 (전체 페이지 재요청)  | 빠름 (필요한 부분만 업데이트)|
| SEO              | 용이                 | 취약할 수 있음              |
| 서버 부하         | 높음                 | 낮음                        |

## RESTful 아키텍처
`RESTful`이란 웹 애플리케이션에서 **리소스를 중심으로 설계된 API 방식**으로, REST(Representational State Transfer) 아키텍처 스타일을 따르는 웹 서비스이다. RESTful한 시스템은 URL을 통해 자원을 표현하고, HTTL 메서드(GET, POST, PUT, DELETE 등)를 통해 자원에 대한 행위를 명확하게 구분한다. 예를 들어 /users는 사용자 목록이라는 자원을 의미하고, 여기에 `GET` 요청을 보내면 목록을 조회하고, `POST`는 새 사용자를 생성하며, /users/1에 `PUT` 요청을 보내면 1번 사용자를 수정하는 방식이다.
RESTful API는 구조가 직관적이고 일관성이 있어 클라이언트와 서버 간의 통신이 명확하고 유지보수에 용이하다.

### REST의 핵심 구성 요소
#### 자원(Resource)
REST에서 모든 것은 **자원**이다.

### HTTP 메서드

| HTTP 메서드 | RESTful 행위 (CRUD) | 설명                                      |
|:----------:|------------------|-----------------------------------------|
| GET        | Read (조회)       | 자원을 조회합니다.                         |
| POST       | Create (생성)     | 새로운 자원을 생성합니다.                  |
| PUT/PATCH  | Update (수정)     | 자원을 수정합니다. (PUT은 전체 수정, PATCH는 부분 수정) |
| DELETE     | Delete (삭제)     | 자원을 삭제합니다.                         |

### RESTful URL 예시

| 목적             | 비(非) RESTful 스타일    | RESTful 스타일      |
|----------------|-----------------|----------------|
| 사용자 목록 조회    | GET /getUsers     | GET /users      |
| 사용자 생성        | POST /createUser  | POST /users     |
| 100번 사용자 조회   | GET /getUserInfo?id=100 | GET /users/100  |
| 100번 사용자 삭제   | GET /deleteUser/100      | DELETE /users/100 |

## Route(라우트)
라우트(Route)는 웹 애플리케이션에서 클라이언트가 요청한 URL 경로와 HTTP 메서드(GET, POST 등)에 따라 서버가 어떤 동작을 수행할지를 정의하는 규칙이다. 즉, 사용자가 /login, /about같은 특정 경로로 요청을 보냈을 때, 그 요청을 처리할 코드를 지정해주는 역할을 한다. Express에서는 app.get(), app.post() 같은 메서드를 통해 라우트를 설정하며, 각 라우트에는 요청 정보를 처리하고 응답을 보내는 핸들러 함수가 연결된다. 라우팅은 웹 서버가 어떤 페이지나 데이터를 제공할지를 결정하는 핵심적인 구조이다.

### route 메서드
Express의 .route() 메서드는 동일한 URL 경로에 대해 여러 HTTP 메서드(GET, POST, PUT 등)를 체이닝 방식으로 한 번에 정의할 수 있게 해주는 기능이다. 예를 들어 /user라는 경로에 대해 GET, POST, DELETE 요청을 따로따로 app.get, app.post 등으로 작성하지 않고, app.route('/user')로 묶어서 관리하면 코드가 더 간결하고 구조화되어 유지보수에 유리하다.
